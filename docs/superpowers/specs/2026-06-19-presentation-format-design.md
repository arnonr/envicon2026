# Presentation Format Field — Design Spec

Date: 2026-06-19

## Goal

Add a required "รูปแบบการนำเสนอ" (presentation format) field to the submission form with two options: แบบบรรยาย (oral) and โปสเตอร์ (poster). Shown to all submitters, persisted with versioning, and displayed in every place `submitterType` and `educationLevel` are shown.

## Scope

1. New `presentationFormat` column on `submissions` and `submission_versions`
2. Required field on the submission form, persisted via POST/PUT/revise
3. Display in dashboard, admin list, both detail modals (with version diff in admin modal)
4. Migration with backfill of existing rows
5. Backend API projections in `admin.ts` and the admin-reviews snapshot path (lessons learned from the education-level final review)

## Section 1: Data Model

### Schema (`backend/src/db/schema.ts`)

Add to `submissions` table (immediately after `educationLevel`):
```ts
presentationFormat: mysqlEnum("presentation_format", ["oral", "poster"]).notNull(),
```

Add the same field to `submission_versions` (snapshot, like `submitterType` and `educationLevel`):
```ts
presentationFormat: mysqlEnum("presentation_format", ["oral", "poster"]).notNull(),
```

The field is **NOT NULL with no default** — the application enforces the value at create time. Existing rows are backfilled to `"oral"` in the migration (oral is the most common presentation mode for academic conferences; flag during review if poster should be the default instead).

### Migration

The project uses Drizzle migrations in `backend/drizzle/`. The new field's migration file (generated by `cd backend && bun run db:generate`) will add the column. To avoid `db:push` failing on existing rows where the column is added as NOT NULL, the generated migration must be edited to the 3-step pattern (nullable-add → backfill → enforce NOT NULL) for both tables:

1. `ALTER TABLE submissions ADD COLUMN presentation_format enum('oral','poster') NULL;`
2. `UPDATE submissions SET presentation_format = 'oral' WHERE presentation_format IS NULL;`
3. `ALTER TABLE submissions MODIFY COLUMN presentation_format enum('oral','poster') NOT NULL;`
4-6. Same 3 steps for `submission_versions`

Apply: `cd backend && bun run db:push`. Includes Drizzle meta files (`meta/0005_snapshot.json`, `meta/_journal.json`).

## Section 2: Backend API

### `backend/src/routes/submissions.ts`

**POST `/submissions`** — add required `presentationFormat` to body validation and insert:
```ts
presentationFormat: t.Union([t.Literal("oral"), t.Literal("poster")]),
```

**PUT `/submissions/:id`** — add the same field as optional:
```ts
presentationFormat: t.Optional(t.Union([t.Literal("oral"), t.Literal("poster")])),
```
And add `...(body.presentationFormat && { presentationFormat: body.presentationFormat })` to the update set.

**POST `/submissions/:id/revise`** — snapshot the field into the new `submission_versions` row:
```ts
presentationFormat: sub.presentationFormat,
```

### `backend/src/routes/admin-reviews.ts`

In `ensureCurrentSubmissionVersion`, add `presentationFormat: submission.presentationFormat` to the `submissionVersions` insert. **This is a pre-emptive fix** to prevent the same type-check failure that the education-level feature encountered in Task 8 — when an admin triggers a new version, the snapshot must include all snapshot-able fields, not just the ones we remember to add.

### `backend/src/routes/admin.ts`

The admin list and Excel export endpoints use **explicit column projections** (`.select({...})`). Education level was added in the schema but not in these projections in the original implementation, which the final code review caught as a critical bug. To avoid the same mistake, update both projections in the same commit that adds the field.

**GET `/submissions`** select (line ~113-128) — add `presentationFormat: submissions.presentationFormat` after `educationLevel`.

**GET `/submissions/export`** select (line ~200-218) — same addition.

**Excel export mapping** — add a Thai label map (mirroring `EDUCATION_LEVEL_NAMES`):
```ts
const PRESENTATION_FORMAT_NAMES: Record<string, string> = {
  oral: "แบบบรรยาย",
  poster: "โปสเตอร์",
};
```

Add a worksheet column `{ header: "รูปแบบการนำเสนอ", key: "presentationFormat", width: 18 }` after the education-level column, and `presentationFormat: PRESENTATION_FORMAT_NAMES[submission.presentationFormat] ?? submission.presentationFormat` to the row mapper. Bump the autoFilter range to `A1:T1` (one more column).

## Section 3: Frontend Form

### `frontend/components/submission/SubmissionForm.vue`

Add to `SubmissionFormData` interface:
```ts
presentationFormat: string;
```

Add options constant:
```ts
const PRESENTATION_FORMAT_OPTIONS = [
  { label: 'แบบบรรยาย', value: 'oral' },
  { label: 'โปสเตอร์', value: 'poster' },
];
```

Add a `URadioGroup` (not `USelect` — only 2 options, radio is more readable like `submitterType`):
```vue
<UFormGroup label="รูปแบบการนำเสนอ" required>
  <URadioGroup
    :model-value="modelValue.presentationFormat"
    :options="PRESENTATION_FORMAT_OPTIONS"
    @update:model-value="update('presentationFormat', $event as string)"
  />
</UFormGroup>
```

Place immediately after the `educationLevel` field (groups submitter info together).

### `frontend/pages/submit/index.vue`

- Initial form: `presentationFormat: ''`
- `isStep1Valid`: add `|| !f.presentationFormat` to the falsy chain
- POST body: add `presentationFormat: form.value.presentationFormat`

### `frontend/pages/submissions/[id]/revise.vue`

- Form init: `presentationFormat: ''`
- Validation: include `!values.presentationFormat` in the chain
- After loading submission, prefill `educationLevel`-style with `presentationFormat: submission.presentationFormat`
- PUT body: include `presentationFormat: form.value.presentationFormat`

## Section 4: Display Layer

### Label helper

Inline in each consumer (mirroring the education-level spec's decision — 4 trivial call sites, no composable):
```ts
function presentationFormatLabel(v: string): string {
  return ({ oral: 'แบบบรรยาย', poster: 'โปสเตอร์' } as Record<string, string>)[v] ?? v;
}
```

### `frontend/pages/dashboard/index.vue`

- `Submission` interface: add `presentationFormat: string` after `educationLevel`
- Helper after `educationLabel`
- Append `· {{ presentationFormatLabel(sub.presentationFormat) }}` to the list item meta line

### `frontend/components/submission/SubmissionDetailModal.vue` (author modal)

- `Submission` interface: add `presentationFormat: string`
- Helper after `educationLabel`
- Add a `<div>` cell in the info grid, directly after the `educationLevel` cell

### `frontend/pages/admin/index.vue` (admin list)

- `Submission` interface: add `presentationFormat: string`
- Helper after `educationLabel`
- Add a `<th>รูปแบบการนำเสนอ</th>` column header after the education-level column
- Add a `<td>` cell with `{{ presentationFormatLabel(sub.presentationFormat) }}` after the education-level cell

### `frontend/components/admin/SubmissionDetailModal.vue` (admin modal)

- `Submission` interface: add `presentationFormat: string`
- `SubmissionVersion` interface: add `presentationFormat: string`
- Helper after `educationLabel`
- Add a `<div>` cell in the info grid, directly after the `educationLevel` cell
- Add a row in `comparisonRows` (version diff) — placed after the education-level entry:

## Out of Scope

- Different file upload requirements per format (e.g., poster PDF vs oral paper)
- Different reviewer rubrics per format
- Different fee calculation per format
- Conditional validation tied to the format
- Schedule or room-allocation logic
- Reviewer-side or notification-side use of this field
